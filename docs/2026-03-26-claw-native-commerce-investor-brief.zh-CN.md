# Claw Native 电商架构简版说明

**适用对象：** 投资人、合作方、外部高层沟通  
**版本基线：** `codex/openclaw-web-validation`，当前实现态

## 1. 这家公司在做什么

Claw Native 不是在做“另一个电商 App”，也不是在做“会推荐商品的聊天机器人”。

它在做的是一套 **买方代理优先的消费决策与交易执行系统**：

- 用户不需要自己逐个平台比较、筛选、凑单、判断卖家和时效
- Claw 先理解用户当前的购买任务
- 然后以结构化代理链路完成询价、比选、策略判断、锁库存和提交订单
- 最后把整条决策链路解释给用户和运营方看

一句话说，Claw Native 想把“人自己购物”升级成“代理替人完成购物”。

## 2. 为什么这个方向成立

传统电商优化的是：

- 更多流量
- 更多停留
- 更多点击和转化

Claw Native 优化的是：

- 更少决策时间
- 更少认知负担
- 更高比例的任务级成交

这意味着它不是内容平台逻辑，而是 **代理交易逻辑**。

在这个逻辑里，核心资产不再只是流量，而是：

- 用户长期消费偏好
- 用户决策模式
- 可执行的 seller protocol
- 全链路 explanation 和 trust

## 3. 当前系统已经实现了什么

当前代码已经不是 PPT 架构，而是一个可运行的 MVP 基线，包含两层：

### 第一层：购物副驾

- 以浏览器扩展运行在京东商品页和购物车页
- 帮用户做“怎么买”的决策，而不是泛泛推荐
- 当前重点验证城市白领日用品复购场景

### 第二层：Native Commerce 后端

- 有 buyer API 作为买方代理入口
- 有 procurement orchestrator 作为交易编排核心
- 有 seller-sim 作为卖家网络模拟层
- 有 Web validation console 展示真实链路

这意味着 Claw Native 当前已经能展示一条完整链路：

```text
用户/演示页面提出补货任务
-> buyer API 接收结构化请求
-> orchestrator 生成采购 intent
-> seller-sim 返回多个 seller quote
-> 系统对 offer 排序并选择
-> 经过 policy 判断
-> hold inventory
-> commit order
-> 返回 explanation 和 snapshot
```

## 4. 它和传统导购工具有什么本质差异

传统导购工具通常只做到：

- 比价
- 推荐
- 内容种草
- 跳转成交

Claw Native 的目标是更深一层：

- 从“给建议”进入“替你完成决策”
- 从“商品推荐”进入“任务执行”
- 从“页面操作”进入“协议化交易”

所以它不是 `shopping content`，而是 `procurement agent infrastructure`。

## 5. 为什么这套架构重要

这套架构的价值不只是当前 demo 能跑，而是它定义了未来的扩展方向。

如果系统继续往下演进，它天然可以扩展到：

- 更多消费品类
- 更多 seller network
- 更强的个人偏好和预算管理
- 更高比例的自动成交
- 面向 B 端的采购 agent 输出

也就是说，今天的 MVP 虽然从消费和购物切入，但底层实际上是在验证一个更大的问题：

**代理能不能成为新的交易入口。**

## 6. 当前最关键的技术护城河

从当前实现来看，真正的护城河不是“模型参数”，而是下面几件事能不能同时成立：

1. 代理能理解用户场景，并把它转换成结构化交易请求
2. 系统能在 seller 之间做可解释的自动比选
3. 所有自动决策都能被审计、回放和纠偏
4. 交易执行有明确编排核心，而不是散落在前端逻辑中

这也是为什么当前架构里，`orchestrator` 比一个聊天界面更关键。

## 7. 现在还没有做完什么

当前系统仍然是 MVP，不应该被包装成完整平台。

还没有完成的关键项包括：

- 真实外部 seller network 接入
- 真实支付与履约系统
- C 端扩展和交易后端的正式联通
- 生产级持久化交易底座
- 更复杂的个性化消费理解

所以当前更准确的说法是：

**Claw Native 已经把“代理式电商”的最小闭环跑通了，但还没有进入生产级商业化阶段。**

## 8. 你可以如何理解它的阶段位置

如果用创业语言来描述，当前阶段不是“已经做成一家电商公司”，而是：

- 已经找到方向：消费决策代理
- 已经找到入口：购物副驾
- 已经找到后端骨架：buyer API + orchestrator + seller protocol
- 已经证明概念：可以跑结构化的 agent commerce 链路

这意味着下一阶段的重点，不是再做更多前台页面，而是：

- 接入更真实的 seller 生态
- 打通前端副驾和后端代理系统
- 让更多交易真正由代理完成，而不是只由页面展示

## 9. 结论

Claw Native 当前电商架构的核心判断可以概括成一句话：

**它不是要做一个更会推荐商品的入口，而是要成为用户默认的消费决策代理和交易执行代理。**

今天这个系统已经具备：

- 面向用户的购物副驾入口
- 面向交易的结构化编排后端
- 面向卖家的协议化交互层
- 面向信任的 explanation 与审计能力

这就是它最值得关注的地方。

# Claw Native 路演口径版

**适用对象：** 融资路演、战略合作沟通、外部高层介绍  
**用途：** 帮你把 Claw Native 讲成“代理式电商基础设施”，而不是“另一个 AI 导购产品”

## 1. 30 秒版本

Claw Native 不是在做一个更会推荐商品的电商入口，而是在做用户默认的消费决策代理。

今天用户买东西，最大的成本不是支付，而是决策本身：选什么、比什么、怎么买、是否可信、是否值得现在下单。  
Claw Native 把这件事从“用户自己做”变成“代理替你做”，并且不是只给建议，而是把需求转成结构化采购流程，完成 seller 比选、策略判断、锁库存、提交订单和全链路 explanation。

所以我们做的不是一个新商城，而是一套 buyer-agent-first 的 commerce stack。

## 2. 3 分钟版本

### 2.1 问题

传统电商优化的是流量和转化，但用户真正痛的不是“没有商品”，而是“完成购买任务太费脑”。

尤其在高频消费和复购场景里，用户每次都要重复做这些事：

- 选规格
- 比卖家
- 算时效
- 看价格
- 判断值不值得现在买

这套流程本身就是一个代理问题。

### 2.2 我们的判断

未来电商会从 `App + 搜索 + 推荐` 走向 `Agent + 协议 + 编排`。

也就是说，用户不会总是自己浏览页面、逐步点击完成购买，而是会越来越多地把购买任务交给自己的主代理。

Claw Native 就是在做这个主代理的消费和交易能力。

### 2.3 当前产品形态

我们已经做出两层系统：

第一层是购物副驾。  
它以浏览器扩展的形式运行在京东商品页和购物车页，帮助用户做“怎么买”的决策。

第二层是 Native Commerce 后端。  
它接收结构化购买请求，通过 buyer API 进入 procurement orchestrator，再连接 seller protocol、seller selection、policy engine、checkout 和 explanation。

所以我们不是停留在推荐层，而是已经进入了交易执行层。

### 2.4 当前证明了什么

当前系统已经能跑通一条真实代理链路：

```text
结构化补货请求
-> 生成采购 intent
-> 获取多个 seller quote
-> 自动排序选择
-> policy 判断
-> hold inventory
-> commit order
-> 返回 explanation
```

这件事的意义是，我们已经证明：

- 代理可以成为交易入口
- 交易可以被结构化编排
- 自动决策可以解释和回放

## 3. 10 分钟版本

## 3.1 为什么这不是导购工具

导购工具解决的是“看什么”，Claw Native 解决的是“完成购买任务”。

导购的尽头通常是：

- 推荐更准
- 内容更丰富
- 转化更高

Claw Native 的尽头不是内容，而是：

- 更高比例的用户任务被代理完成
- 更高比例的交易由代理直接执行
- 更少的人类手工比较和点击

这两者不是一个品类。

## 3.2 为什么这个架构重要

我们非常强调架构，是因为在 agent commerce 里，前端体验不是全部，真正的护城河在后端交易能力。

当前系统已经明确建立了几个关键层：

- buyer API：作为代理交易入口
- orchestrator：作为交易大脑
- seller protocol：作为卖家边界
- offer evaluator：作为 seller 选择机制
- policy engine：作为风险和授权边界
- memory / explanation：作为信任基础

这意味着 Claw Native 不是“把 AI 塞进电商前台”，而是从底层重构交易流程。

## 3.3 为什么我们先从消费和购物切入

因为消费和购物有 3 个很强的优势：

1. 高频
2. 决策成本高
3. 交易闭环清晰

这是最适合训练用户把一部分决策权交给代理的场景。

一旦这个信任关系建立，后续可以自然扩展到：

- 家庭采购
- 本地生活
- 企业采购
- 更复杂的多 agent 协同交易

## 3.4 当前还没做什么

我们不会把现在的系统夸大成已经商业化完成的平台。

现在还没完成的包括：

- 真实外部 seller network
- 生产级订单存储
- 真实支付与履约
- C 端副驾和后端交易系统的正式打通

所以当前阶段更准确的说法是：

我们已经把代理式电商的最小架构闭环跑通，并正在把它从 demo 形态推向真实网络。

## 4. 你在路演里可以反复强调的三句话

### 4.1 不是“AI 推荐”，是“AI 代理”

我们不是让 AI 告诉你什么值得买，而是让 AI 替你完成越来越多的购买决策。

### 4.2 不是“内容分发”，是“任务执行”

我们的核心不是让用户停留更久，而是让用户更快完成购买任务。

### 4.3 不是“流量平台”，是“交易操作系统”

长期看，真正有价值的位置不是一个新的前台流量入口，而是用户主代理背后的交易底座。

## 5. 常见问题与回答口径

## 5.1 这是不是一个 AI 导购插件

不是。浏览器扩展只是当前入口层。  
真正的系统重点在 buyer API、orchestrator、seller protocol 和 explanation。

## 5.2 这是不是要自己做电商平台

不一定。长期更有价值的是做交易编排和代理执行层，而不是重做一个流量平台。

## 5.3 为什么现在要做 seller-sim

因为没有 seller protocol 的本地模拟，就无法验证代理交易链路。  
seller-sim 的作用不是替代真实商家，而是把系统边界先定义清楚。

## 5.4 为什么现在强调 explanation

因为一旦代理进入交易执行，信任就不再来自“推荐好不好看”，而来自“为什么这么做、出了问题谁负责、能不能回放”。

## 6. 路演时的顺序建议

推荐顺序是：

1. 先讲问题：今天购物的核心成本是决策，而不是支付
2. 再讲判断：交易入口会从 App 迁移到 Agent
3. 再讲系统：Claw Native 用 buyer-agent-first 的方式重构交易
4. 再讲当前进展：已经跑通结构化交易最小闭环
5. 最后讲扩展：从消费购物切入，向更大的 agent commerce 走

## 7. 收尾句

如果你只想在最后留下一句话，建议用这句：

**Claw Native 不是在做一个更聪明的导购产品，而是在做用户主代理时代的消费交易底座。**
